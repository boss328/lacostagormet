# 06.1 — Migration From BigCommerce

## What we're migrating

| Item | From | To | Priority |
|---|---|---|---|
| Customers | BC export | Supabase `auth.users` + `customers` | HIGH |
| Addresses | BC customer export | `addresses` | HIGH |
| Order history | BC export | `orders` + `order_items` | HIGH (grandpa specifically wants this) |
| Products | BC export | Supabase `products` | MEDIUM — **vendor spreadsheets are the better source** |
| Product images | BC CDN | Supabase Storage | HIGH |
| URL slugs | BC product URLs | 301 redirects in Next.js middleware | CRITICAL (SEO) |
| Blog posts | BC blog | Skip — not valuable | LOW |
| Saved payment methods | Auth.net CIM | Auth.net CIM (unchanged) | AUTO — already there |

## How to get the data out of BigCommerce

Once grandpa gives you admin access:

### Products export
1. `Admin → Products → View`
2. Top-right → Export
3. Template: **Bulk Edit**
4. Format: **CSV**
5. Export → download from Export History

Save to `references/bigcommerce/products-export.csv`.

### Customers export
1. `Admin → Customers → View`
2. Export → all customers, CSV
3. Columns needed: email, first name, last name, phone, company, addresses

Save to `references/bigcommerce/customers-export.csv`.

### Orders export
1. `Admin → Orders → View`
2. Filter: all time, status = any
3. Export → CSV (BigCommerce may split into multiple files if large)

Save to `references/bigcommerce/orders-export.csv` + any order-items CSV.

### URL map
1. Get products-export.csv (has the URL path field for each product)
2. Plus category URLs from the category tree (manually listable, ~30 of them)
3. Plus brand URLs (~15)
4. Plus system pages (About, Contact, Shipping, Privacy, Terms — match these exactly by path)

## Migration scripts (build these in `/scripts/`)

### `import-bc-customers.ts`
```
read: references/bigcommerce/customers-export.csv
for each row:
  1. Create auth.users row via Supabase Admin API with random temp password
  2. Mark needs_password_reset = true
  3. Insert customers row with legacy_bc_customer_id + migrated_from_bc = true
  4. Insert addresses rows for each address on file
  5. Send password-reset email via Resend (templated "Welcome to the new site")
```

Run this ONCE before launch, after all customers are exported. Rate-limit the password-reset emails (throttle 50/sec) so you don't trip spam filters.

### `import-bc-orders.ts`
```
read: orders-export.csv + order-items export
for each order:
  1. Look up customer by email (may match existing or may be missing — that's OK, set customer_id to null for orphan orders)
  2. Insert orders row with:
     - legacy_bc_order_id
     - status = 'paid' (assume; migrated orders are historical)
     - fulfillment_status = 'delivered' (historical)
     - total, subtotal, tax, shipping copied as-is
     - shipping_address, billing_address as jsonb
  3. Insert order_items rows:
     - product_id = lookup by SKU (may be null if SKU not in new catalog)
     - product_name, product_sku, unit_price, quantity, line_subtotal snapshotted from old data
  4. DO NOT re-charge any card
  5. DO NOT create payment rows (or create them with type='migrated' and amount=total so financial views don't double-count)
```

**Important:** mark migrated orders clearly. A filter in admin `/admin/orders` should let grandpa hide pre-migration orders when he wants to see "just the new site's orders."

### `import-vendor-catalog.ts`
```
input: a vendor spreadsheet + vendor_id
for each row:
  1. Normalize column names via saved mapping for this vendor
  2. Match existing product by SKU
  3. If match: update wholesale_cost, possibly other fields per "update policy" setting
  4. If no match: create new product row with retail_price = wholesale_cost * 1.5 (placeholder, grandpa adjusts)
  5. Set preferred_vendor_id = this vendor
  6. Log audit entry
  7. Report: N created, N updated, N skipped, N conflicts
```

### `generate-redirect-map.ts`
```
read: products-export.csv, category list, brand list
build: a redirect object where source_path → destination_path
output: src/middleware/redirects.json (or a TypeScript const) used by middleware.ts
```

**Strategy:**
- **Product URLs:** keep the slug identical. Source = `/old-product-slug/`, destination = `/product/old-product-slug`. Because current BC has flat URLs (no /product/ prefix) and new structure uses `/product/[slug]`.
  - **Alternative:** keep FLAT URLs on new site too. Products live at `/[slug]` with category at `/shop/[category]`. **This is better for SEO continuity.** Downside: tiny risk of slug collision between products and categories. Mitigation: namespace check in middleware.
- **Category URLs:** `/teas/` → `/shop/teas` (or keep at `/teas/` — whichever is cleaner).
- **Brand URLs:** `/brands/big-train/` → `/brand/big-train`
- **System pages:** `/contact-us/`, `/blog/`, `/privacy/`, etc. — these match exactly, just rebuild them.

**Recommendation:** preserve flat URLs. Less redirect work, less SEO risk. Use middleware only for cases that genuinely changed.

## middleware.ts (301 redirects)

```typescript
// src/middleware.ts
import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';
import redirects from './redirects.json';

export function middleware(request: NextRequest) {
  const path = request.nextUrl.pathname;
  
  // Exact match redirects
  if (redirects[path]) {
    return NextResponse.redirect(
      new URL(redirects[path], request.url),
      { status: 301 }
    );
  }
  
  // Handle trailing slash inconsistency (BC URLs had trailing /)
  if (path.endsWith('/') && path.length > 1) {
    const noSlash = path.slice(0, -1);
    if (redirects[noSlash]) {
      return NextResponse.redirect(new URL(redirects[noSlash], request.url), { status: 301 });
    }
  }
  
  return NextResponse.next();
}

export const config = {
  matcher: [
    '/((?!api|_next/static|_next/image|favicon.ico|images).*)',
  ],
};
```

## SEO preservation checklist

- [ ] All product URLs return 200 (either direct or via 301)
- [ ] All category URLs return 200
- [ ] All brand URLs return 200
- [ ] Sitemap.xml generated dynamically at `/sitemap.xml` with every product, category, brand, page
- [ ] robots.txt references the new sitemap
- [ ] Schema.org markup on every product page (Product schema with offer, brand, image, name, description)
- [ ] Schema.org Organization markup on homepage
- [ ] Schema.org BreadcrumbList on category + product pages
- [ ] Canonical tags on every page
- [ ] OpenGraph tags for social sharing
- [ ] Google Search Console: verify new site with DNS TXT record or HTML file, submit new sitemap, monitor coverage after launch
- [ ] Core Web Vitals: LCP < 2.5s, CLS < 0.1, FID < 100ms (Next.js SSR helps a lot here)
- [ ] Image optimization: Next.js `<Image>` with `width`/`height`, proper `alt` text

## Product images migration

BigCommerce serves images from `cdn11.bigcommerce.com/s-u3ny6186xw/...`. These URLs will stop working if grandpa's BC account is cancelled. Migrate them.

**Script `download-bc-images.ts`:**
1. For each product in BC export, download each image URL
2. Upload to Supabase Storage bucket `product-images/`
3. Insert `product_images` row with new Supabase URL
4. Mark first image as `is_primary = true`

Parallelize with a concurrency limit (10 at a time). Log failures for manual review.

## Post-launch BigCommerce plan

- **Don't cancel BC immediately.** Keep it live for 30-90 days as a read-only archive.
- After confirmation that 301s are working and no critical data was missed, grandpa can downgrade or cancel.
- Full BC data backup (export all CSVs + download all images) stored in grandpa's Google Drive as insurance.
