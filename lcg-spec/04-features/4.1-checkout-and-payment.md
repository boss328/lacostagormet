# 04.1 — Checkout & Payment Flow

## Overview

Use **Authorize.net Accept.js (hosted card fields)** so card data never touches the server. This keeps PCI compliance at SAQ A-EP instead of SAQ D. No exceptions.

## Checkout pages

```
/cart               → view cart, update quantities, proceed to checkout
/checkout           → 3-step checkout (single-page, collapsible)
/checkout/confirmation  → post-purchase thank-you page
```

### Single-page checkout sections

1. **Contact** — email (creates account on purchase if guest)
2. **Shipping** — address form or select saved address
3. **Shipping method** — calculated based on cart subtotal + destination
   - Continental US ≥ $70: "Free Ground Shipping" ($0)
   - Continental US < $70: "Ground Shipping" ($12.99 — pulled from `settings`)
   - HI/AK: "Ground to Hawaii/Alaska" ($12.99 + $25 = $37.99)
4. **Payment** — Accept.js iframe OR saved payment method
5. **Review & place order** — itemized total, tax, shipping, grand total

### Business checkout modifier

If user checks "This is for my business":
- Show fields: company name, tax ID (optional, for future tax-exempt flow)
- Save to order's `is_business`, `business_name`, `business_tax_id`

## Authorize.net AFDS handling

Grandpa has **5-6 fraud rules** configured in his Auth.net merchant account dashboard. These live server-side at Auth.net and apply automatically to every transaction submitted through his account — no app code needs to enforce them.

Common AFDS rules he likely has:
- AVS mismatch hold
- High-risk country block
- IP velocity filter
- Transaction amount threshold
- Shipping/billing address mismatch
- CVV mismatch

When a rule triggers, Auth.net returns one of:
- `responseCode: 1` with a fraud filter message → held for review
- `responseCode: 4` → held for review (dedicated code)
- `responseCode: 2` → declined (could be fraud or bank-level)

**Code must handle all three:**

```typescript
// pseudo
if (response.responseCode === '1' && response.messages[0].text.includes('fraud')) {
  order.status = 'payment_held';
  payment.fraud_held = true;
  payment.fraud_reason = response.messages[0].text;
} else if (response.responseCode === '4') {
  order.status = 'payment_held';
  payment.fraud_held = true;
} else if (response.responseCode === '2') {
  order.status = 'cancelled';
  payment.status = 'failed';
} else if (response.responseCode === '1') {
  order.status = 'paid';
  payment.status = 'succeeded';
}
```

## Webhook handler at `/api/webhooks/authnet`

Auth.net sends webhooks for:
- `net.authorize.payment.fraud.held` — order was held by AFDS
- `net.authorize.payment.fraud.approved` — admin approved a held transaction
- `net.authorize.payment.fraud.declined` — admin declined a held transaction
- `net.authorize.payment.authcapture.created` — payment captured
- `net.authorize.payment.refund.created`
- `net.authorize.payment.void.created`

Signature verification: use the `AUTHNET_SIGNATURE_KEY` to verify X-ANET-Signature header on every webhook. Reject anything unverified.

On each webhook:
1. Verify signature
2. Look up local order by `authnet_transaction_id`
3. Update order/payment status atomically
4. Insert `audit_log` row
5. If status changed to `paid`: trigger customer confirmation email + admin notification

## Customer-facing error handling

**Declined card:** generic "Your payment could not be processed. Please try a different card or contact your bank." Do not expose Auth.net response codes to the customer — they're not helpful and can be abused.

**Held for review:** friendly message on `/checkout/confirmation`:

> "Your order has been received! We're doing a quick manual review of your payment — you'll hear from us within 1 business day. Order number: LCG-XXXXX"

Also send them an email:

> "Thanks for your order. Our team is doing a quick security review on this purchase, and you'll get a confirmation email once that's complete — usually within a business day. If there's any issue we'll reach out directly."

## Saved payment methods (CIM tokenization)

On checkout, if "Save this card for next time" is checked:
1. After successful auth, call Auth.net CIM API to create a customer profile + payment profile
2. Store `authnet_profile_id` and `authnet_payment_profile_id` in `saved_payment_methods`
3. Store only: last 4, brand, expiry — never PAN or CVV

On subsequent checkouts, customer can select the saved method; server uses the stored profile IDs to charge it.

## Tax calculation

**V1: zero tax by default.** Jeff's catalog is all non-taxable grocery food in California (his only nexus state). The tax calc function returns `0` unless:
- `settings.tax.enabled` is `true`, AND
- Cart contains a product with a non-null `tax_class`, AND
- Ship-to state is in `settings.tax.nexus_states`

**Implementation:**
- Call `calculateTax(cart, shipping_address)` at checkout step 5 (review)
- Returns `{ amount: 0, breakdown: [] }` for all typical orders
- Store the calculated tax in `orders.tax` (will be `0.00` for all v1 orders under normal conditions)
- No external API call, no caching needed

**Phase 2:** if Jeff adds taxable SKUs or hits nexus in a new state, implement TaxJar via the same function signature. See `05-integrations/5.1-integrations.md` for the upgrade path.

## Shipping calculation

- Entirely rule-based, pulled from `settings`:
  ```ts
  if (state === 'HI' || state === 'AK') {
    shipping = 12.99 + 25;  // always charge base rate + surcharge
  } else if (subtotal >= 70) {
    shipping = 0;
  } else {
    shipping = 12.99;
  }
  ```
- Later v2: real-time rates via Shippo with origin = vendor warehouse (requires knowing which vendor ships what)

## Guest vs. account checkout

- Both supported
- Guest order: store email, no `customer_id`. If that email later registers, merge their order history.
- Account checkout: auto-populate shipping/billing from defaults, skip straight to payment if a saved card exists
