# 04.2 — Multi-Vendor Fulfillment (Email-Based)

## The model, simplified

After talking with Jeff, the fulfillment model is genuinely simple:

1. **Each product has one `supplier` field** (a dropdown in admin). Options: Houston's, Sunny Sky, Mocafe, David Rio, Monin, Torani, Kerry, DaVinci Gourmet, iBev, Smartfruit, Tiki Breeze, or **Home/Garage** for things Jeff ships himself.
2. **When an order comes in**, the admin groups line items by their supplier. Each group becomes a "vendor order."
3. **For supplier groups** (Houston's, Sunny Sky, etc.): admin generates a **pre-filled email** Jeff can edit, then sends from his Google Workspace account.
4. **For Home/Garage groups**: they land in a **"Ship Yourself"** queue. No supplier email. Jeff handles labels his current way (USPS/UPS/PayPal at home). Marks shipped + pastes tracking when done.

**Jeff explicitly said: keep it simple.** No per-warehouse routing. No multiple vendor API integrations. One supplier per product, one email per order group, done.

## Product → supplier assignment

- Each `product` has a `preferred_vendor_id` FK to `vendors`
- When Jeff adds or edits a product, the Supplier dropdown lists all active vendors (including Home/Garage)
- Orders auto-populate `order_items.assigned_vendor_id` from the product's `preferred_vendor_id` at order time
- Admin order view lets Jeff re-assign per line item before sending the email (e.g., Houston's is out of stock → reassign this line to "Home/Garage" to ship himself)

## Email composition (for non-self-fulfilled vendors)

When Jeff clicks "Send to [Vendor]":

1. System loads the vendor's `email_template` from the `vendors` table (Jeff writes this once, per vendor, in admin settings)
2. Substitutes variables with actual order data (see Variables below)
3. Shows a preview modal — Jeff can edit body, subject, PO number before sending
4. "Send" button sends the email via Gmail API from Jeff's real Google Workspace address
5. Stores email body + subject + Gmail message ID on the `vendor_orders` row
6. Updates `vendor_orders.status = 'sent'` and `email_sent_at`
7. Updates `orders.fulfillment_status` to `partially_sent` or `fully_sent`

### Email template variables

When Jeff writes a template, he can use these placeholders (or not — totally up to him):

| Variable | Example value |
|---|---|
| `{{order_number}}` | `LCG-10042` |
| `{{order_date}}` | `April 17, 2026` |
| `{{customer_name}}` | `Jordan Martinez` |
| `{{customer_company}}` | `Cafe Brio` |
| `{{shipping_address}}` | Full multi-line address |
| `{{shipping_address_inline}}` | Single-line address |
| `{{items_table}}` | Pre-formatted table of items in this group |
| `{{items_list}}` | Bullet list of items |
| `{{po_number}}` | The PO number Jeff typed in the form (optional) |
| `{{my_account_number}}` | Jeff's account number with this vendor (stored on vendor row) |

**No label printing integration v1.** If Jeff wants in-admin label printing later (ShipStation/ShipEngine/EasyPost), that's a Phase 2 add-on worth ~$15/mo plus a few hours of integration work.

## Auto-send toggle (default OFF)

Admin setting: `vendor_email.auto_send_enabled` (boolean, default `false`).

When ON: after payment captures, the system automatically sends vendor emails if all products in the order have a supplier set and (crucially) none of them are Home/Garage.

When OFF (the default): Jeff reviews every order and clicks "Send" on each supplier group.

Don't ship with auto-send ON. Jeff will want to review orders for the first several weeks, then flip it on when he trusts the system.

### Auto-send fallback logic

Even with auto-send ON, fall back to manual review for:
- Orders flagged by Auth.net AFDS (held for review)
- Orders where any product is missing `preferred_vendor_id`
- Orders containing Home/Garage items (those always need Jeff's hands)
- Orders over a configurable $ threshold (`settings.auto_send_max_amount`, default $500)

## Tracking v1

Simple paste-back model.

1. Supplier replies to Jeff's email with tracking number
2. Jeff opens the order in admin, pastes tracking into the vendor group's tracking field
3. Carrier dropdown (UPS / FedEx / USPS / other)
4. Checkbox "Notify customer" (default ON)
5. Save → triggers "Your order has shipped" email to customer via Resend with tracking link
6. Status updates: `vendor_orders.status = 'shipped'`, `orders.fulfillment_status = 'partially_shipped'` or `'shipped'`

**No automated carrier polling in v1.** Customers get one shipping email with the tracking link they can click themselves.

**Phase 2 decision:** Jeff will share what his suppliers' APIs / email formats actually look like. If they're standardized, automated tracking (via ShipEngine or similar) is a 4-6 hour build that layers in cleanly. Revisit once we've seen the real supplier data.

## Status progression

```
Order placed → status=paid, fulfillment=unfulfilled
  ↓
Vendor email(s) sent → fulfillment=partially_sent or fully_sent
  ↓
Tracking added (any group) → fulfillment=partially_shipped
  ↓
Tracking added (all groups) → fulfillment=shipped
  ↓
(Optional, manual later) Mark delivered → fulfillment=delivered
```

## Edge cases

- **Houston's is out of stock → Jeff reassigns a line to Home/Garage.** Admin supports re-assigning `order_items.assigned_vendor_id` even after the vendor email is sent. If changed, Jeff can resend or cancel the email.
- **Customer cancels before vendor email sent.** Void the Auth.net auth, cancel order. Clean.
- **Customer cancels after vendor email sent.** Jeff manually emails the vendor to cancel, then marks the order refunded in admin.
- **Jeff wants to add a new supplier.** Admin → Vendors → Add New. Fills in name, email, template. That vendor now appears in the Supplier dropdown on products.

## What Jeff does NOT have to do

- He does not pick warehouses (not tracking them)
- He does not enter tracking numbers into the carrier's website
- He does not manually email customers when something ships
- He does not manage separate inventory buckets per vendor
