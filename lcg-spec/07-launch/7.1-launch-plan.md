# 07.1 — Launch Plan

## The order of operations (don't skip steps)

1. **Build on staging** — `staging.lacostagourmet.com` subdomain on Orange VPS, separate PM2 process on port 3001
2. **Load test data** — import BC customers/products, set up test vendor accounts, seed settings
3. **Smoke test** — full end-to-end: browse, add to cart, check out (with a real card, real Auth.net sandbox or low-$ test), verify email, test admin fulfillment, paste tracking
4. **Grandpa UAT** — walk him through staging, fix issues, repeat until he's comfortable
5. **Pre-launch checklist** — run every item in the checklist below
6. **DNS cutover** — change A record, new site live, old BC site goes dark (redirect BC to lacostagourmet.com if BC supports it)
7. **Post-launch watch** — monitor for 72 hours, especially error rates and checkout success
8. **BigCommerce wind-down** — 30 days later, after confirming stability

## Staging setup

```bash
# on Orange VPS
mkdir -p /var/www/lcg-staging
cd /var/www/lcg-staging
# clone repo, npm install, build
pm2 start ecosystem.config.js --only lcg-staging
```

Staging `.env`:
- `NEXT_PUBLIC_SITE_URL=https://staging.lacostagourmet.com`
- Auth.net `AUTHNET_ENVIRONMENT=sandbox` (use test credentials)
- TaxJar sandbox mode
- Resend: use Resend's testing mode OR send-to-Asher-only
- Supabase: a separate Supabase project for staging data

Nginx subdomain:
```nginx
server {
    server_name staging.lacostagourmet.com;
    location / {
        proxy_pass http://localhost:3001;
        # standard Next.js proxy headers
    }
}
```

## DNS cutover

When everything in the checklist is green:

### Day of cutover (pick a Tuesday, not Friday)

1. **9am PT:** TTL prep. 24 hours before cutover, reduce DNS TTL on lacostagourmet.com A record to 300 seconds (5 min) so DNS propagates fast.
2. **11am PT:** final staging test. Place a real order with a real card. Verify everything.
3. **12pm PT:** switch production.
   - Update A record from BigCommerce IP → Orange VPS IP
   - Update any CNAME records (www → @)
   - Update Nginx on VPS to handle `lacostagourmet.com` and `www.lacostagourmet.com` in addition to staging
   - Verify SSL cert provisions (Let's Encrypt via certbot auto-renewal)
4. **12:05pm PT:** curl the site from a few different networks to verify new DNS is resolving
5. **12:15pm PT:** send password-reset emails to all migrated customers ("Welcome to the new site")
6. **12:30pm PT:** update Google Search Console — add new site, submit new sitemap, request indexing of homepage
7. **1pm PT:** verify Google Merchant Center product feed (if set up) pointing to new site
8. **Rest of day:** monitor error logs, checkout success rate, customer support inbox

### Monitoring for 72 hours after

- Watch Sentry (or server logs) for error spikes
- Watch `/admin/orders` for failed checkouts
- Watch Resend dashboard for email deliverability issues
- Watch Google Search Console for crawl errors
- Watch organic traffic — modest dip is normal (1-2 weeks to recover), major dip means redirects broke

## Rollback plan

If something goes catastrophically wrong within first 24 hours:

1. Revert DNS A record to BigCommerce IP (5-minute TTL means fast rollback)
2. Keep BigCommerce active during this period exactly for this reason
3. Diagnose on staging, re-cut when fixed

After 72 hours, BC goes read-only; after 30 days it can be cancelled.
