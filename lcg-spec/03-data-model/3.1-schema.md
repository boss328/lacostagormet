# 03.1 — Supabase Schema

This is the complete data model. Claude Code should generate migrations from this file; do not invent new tables without updating this doc first.

## Design principles

1. **Snapshot money at transaction time.** Any field that represents money at a point in time (wholesale cost, retail price, fees) must be stored denormalized on the transaction record. Don't join back to `products.price` to compute historical margin — that breaks the moment prices change.
2. **Every table has `id`, `created_at`, `updated_at`.** UUIDs as PKs via `uuid_generate_v4()`. `updated_at` via trigger.
3. **Soft deletes via `deleted_at`** on products, customers, vendors. Never hard-delete.
4. **RLS on every table.** Customers can only see their own rows. Admins see everything. Service role bypasses for server-side operations.
5. **Indexes on FKs and commonly-filtered columns.** Document them in the migration file.

## Tables

### `vendors`
One row per supplier Jeff buys from (or fulfills from himself — "Home/Garage" is a vendor row too, just with a different flag).

```sql
id                uuid PK
name              text NOT NULL              -- "Houston's", "Sunny Sky", "Mocafe", "Home/Garage"
slug              text UNIQUE NOT NULL       -- "houstons", "sunny-sky", "home"
is_self_fulfilled boolean DEFAULT false      -- true for "Home/Garage" row; drives different UI behavior
contact_email     text                       -- nullable for self-fulfilled vendors
contact_name      text
phone             text
email_template    text                       -- Jeff writes this per vendor; nullable
notes             text
display_order     int DEFAULT 0              -- order in dropdowns
is_active         boolean DEFAULT true
created_at        timestamptz
updated_at        timestamptz
deleted_at        timestamptz
```

**Seed on launch:** one row per known vendor (Houston's, Sunny Sky, Mocafe, David Rio, Monin, Torani, Kerry, DaVinci Gourmet, iBev, Smartfruit, Tiki Breeze) plus one "Home/Garage" row with `is_self_fulfilled = true`. Jeff can edit/add more in admin UI.

**No `warehouses` JSON.** Jeff clarified he doesn't want per-warehouse inventory tracking. One supplier = one fulfillment destination.

### `orders`
The primary money record. One row per customer purchase.

```sql
id                uuid PK
order_number      text UNIQUE NOT NULL               -- human-readable, e.g. "LCG-10001"
source            order_source NOT NULL DEFAULT 'website'  -- 'website', 'amazon', 'manual' — Phase 2 ready
source_order_id   text                               -- external order ID if source != 'website' (Amazon order ID, etc.)
customer_id       uuid REFERENCES customers(id)       -- null for guests
customer_email    text NOT NULL                       -- always captured
status            order_status NOT NULL DEFAULT 'pending'
fulfillment_status fulfillment_status NOT NULL DEFAULT 'unfulfilled'

-- money, all snapshotted
subtotal          numeric(10,2) NOT NULL
shipping_cost     numeric(10,2) NOT NULL DEFAULT 0
tax               numeric(10,2) NOT NULL DEFAULT 0
discount          numeric(10,2) NOT NULL DEFAULT 0
total             numeric(10,2) NOT NULL

-- addresses snapshot
shipping_address  jsonb NOT NULL                      -- full structured address
billing_address   jsonb NOT NULL

-- business flag
is_business       boolean DEFAULT false
business_name     text
business_tax_id   text

-- internal notes
admin_notes       text

-- legacy
legacy_bc_order_id text

created_at        timestamptz
updated_at        timestamptz
```

```sql
CREATE TYPE order_status AS ENUM (
  'pending',           -- cart → checkout, not yet paid
  'payment_held',      -- Authorize.net AFDS held for review
  'paid',              -- card captured successfully
  'cancelled',
  'refunded',
  'partially_refunded'
);

CREATE TYPE fulfillment_status AS ENUM (
  'unfulfilled',       -- paid but no vendor email sent
  'partially_sent',    -- some vendor emails sent (multi-vendor orders)
  'fully_sent',        -- all vendor emails sent
  'partially_shipped', -- some tracking received
  'shipped',           -- all items have tracking
  'delivered'
);

CREATE TYPE order_source AS ENUM (
  'website',           -- placed through lacostagourmet.com
  'amazon',            -- Phase 2: synced from Amazon Seller Central
  'manual',            -- placed by admin on behalf of customer (phone orders, etc.)
  'migrated'           -- imported from BigCommerce historical data
);
```

## RLS policies (short version)

- **`customers`**: a user can SELECT/UPDATE their own row. Admins can see all.
- **`orders`**: a customer can SELECT their own orders. Admins can see all.
- **`order_items`, `payments`, `vendor_orders`**: tied to order ownership.
- **`products`, `brands`, `categories`, `vendors`**: public SELECT. Admin-only INSERT/UPDATE/DELETE.
- **`settings`**: admin-only. Public can read a whitelist of non-sensitive keys via a view.
- **`audit_log`**: admin-only SELECT.

Service role key (used server-side) bypasses RLS. Client-side code never uses the service key.

## Indexes

Put at minimum:
- `products (slug)`, `products (brand_id)`, `products (sku)`, `products (is_active) WHERE is_active`
- `orders (customer_id)`, `orders (status)`, `orders (created_at DESC)`
- `order_items (order_id)`, `order_items (product_id)`, `order_items (assigned_vendor_id)`
- `vendor_orders (order_id)`, `vendor_orders (vendor_id)`, `vendor_orders (status)`
- `payments (order_id)`, `payments (authnet_transaction_id)`
- `addresses (customer_id)`
- `saved_payment_methods (customer_id)`
